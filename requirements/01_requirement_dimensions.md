# 需求六维框架

> 任何一个 Discourse 插件需求，都可以用这六个维度完整描述。  
> 缺少任何一个维度，AI 都必须做假设，假设往往是错误的起点。

---

## 维度一：角色定义（WHO）

**问：谁会使用这个功能？**

Discourse 中的标准角色：
- `admin` — 站点管理员，拥有最高权限
- `moderator` — 版主，可管理内容
- `user` — 普通登录用户
- `anonymous` — 未登录访客
- `group member` — 特定用户组成员

**写法示例：**
```
角色：
- admin：创建和管理商品，查看购买记录
- user：浏览商品，发起购买
```

**常见错误：** 只说"用户"，没有区分管理员和普通用户。

---

## 维度二：界面入口（WHERE）

**问：每个角色从哪里进入这个功能？**

Discourse 的界面区域：
- 后台管理界面：`/admin/...`（仅 admin 可见）
- 用户个人页：`/u/用户名/...`
- 主题/帖子页：topic 页面内
- 侧边栏/导航：全局可见区域
- 独立路由页面：自定义 URL 路径

**写法示例：**
```
入口：
- admin：后台 → Plugins 菜单 → "会员商品管理" 子菜单
- user：导航栏 → "会员中心" 链接 → 商品列表页
```

**常见错误：** 说"在后台管理"，但没说具体在哪个菜单或路径。

---

## 维度三：操作流程（HOW）

**问：每个角色的完整操作步骤是什么？**

用有序步骤描述，包含：触发动作 → 显示什么 → 用户做什么 → 结果

**写法示例：**
```
admin 创建商品流程：
1. 点击"新建商品"按钮
2. 弹出表单，包含字段：商品名称、价格、描述、有效期（天数）
3. 填写完成后点击"保存"
4. 商品出现在列表中，状态为"已激活"

user 购买商品流程：
1. 进入商品详情页，看到价格和描述
2. 点击"立即购买"按钮
3. 弹出确认对话框，显示商品名称和价格
4. 确认后显示"购买成功"提示，页面刷新显示会员状态
```

## 维度四：数据模型（WHAT DATA）

**问：这个功能需要存储哪些数据？**

不需要写数据库语法，只需描述"存什么"：

**写法示例：**
```
需要存储：
- 商品信息：名称、价格（分为单位）、描述、有效天数、是否激活
- 购买记录：哪个用户、购买了哪个商品、购买时间、到期时间
- 用户会员状态：当前是否是会员、会员到期时间
```

**常见错误：** 完全不提数据，AI 将自行决定存什么字段，可能不符合实际需要。

---

## 维度五：权限与业务规则（RULES）

**问：有哪些限制、验证和业务逻辑？**

包含：谁能做什么、什么条件下才能操作、数据验证规则

**写法示例：**
```
规则：
- 只有 admin 可以创建、编辑、删除商品
- 未登录用户看到商品但无法购买，点击购买跳转到登录页
- 商品价格必须大于 0
- 用户已是会员期间不能重复购买同一商品（或可以叠加天数，需明确）
- 商品被删除后，已购买的用户会员状态不受影响
```

**常见错误：** 不提重复购买、未登录访问等边界场景，开发时这些往往被忽略。

---

## 维度六：状态与反馈（FEEDBACK）

**问：操作成功或失败时，页面如何响应？**

**写法示例：**
```
反馈：
- 创建商品成功：列表自动刷新，显示绿色"商品已创建"提示
- 购买成功：页面顶部显示"购买成功，会员有效期至 XX-XX-XX"
- 购买失败（如余额不足）：显示红色错误提示"操作失败：xxx"
- 会员到期：用户个人页会员状态区域显示"已过期"标签
```

**常见错误：** 完全不提反馈，AI 可能不加任何提示就完成操作，用户不知道是否成功。

---

## 六维框架速查表

| 维度 | 核心问题 | 关键词 |
|------|---------|--------|
| 角色 | 谁来用 | admin / user / moderator |
| 入口 | 从哪进 | 页面路径、菜单位置、按钮 |
| 流程 | 怎么用 | 步骤1→2→3 |
| 数据 | 存什么 | 字段名、字段含义 |
| 规则 | 有什么限制 | 权限、验证、边界条件 |
| 反馈 | 看到什么 | 成功提示、错误提示、状态变化 |
